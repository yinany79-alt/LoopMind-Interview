# 并行开发指南 — 赛博面试官 MVP

> **本文件读者**:你(主操盘人)+ 前端 coding agent + 后端 coding agent
> **作用**:用 git worktree 让前后端两个 agent **物理隔离地并行开发**,最后在主目录联调
> **配套**:[前端设计.md](./前端设计.md)、[接口规范.md](./接口规范.md)

---

## 0. 核心理念

**两个 coding agent 同时干活,但工作目录物理隔离**。它们看不到对方的代码,只通过 [接口规范.md](./接口规范.md) 这份"合同"来对齐。等各自跑通后,在主目录联调。

```
                  主目录(联调用)
                  /Users/yangyinan.8/Desktop/interview/
                  ├── main 分支(空壳 + 文档)
                  └── mvp/
                      ├── docs/      ← 共享的合同
                      ├── frontend/  ← 联调时合并进来
                      └── backend/   ← 联调时合并进来
                          │
        ┌─────────────────┴─────────────────┐
        ↓                                   ↓
   前端 worktree                       后端 worktree
   feature/frontend 分支               feature/backend 分支
   .worktrees/frontend/                .worktrees/backend/
   - 前端 agent 在这里独立开发        - 后端 agent 在这里独立开发
   - 跑 Mock 模式                     - 跑 hardcoded knowledge
   - 只读 docs/                       - 只读 docs/
```

## 1. 第 0 步:初始化 Git 仓库(主操盘人手动跑一次)

当前 `/Users/yangyinan.8/Desktop/interview/` 还不是 git 仓库,先一次性初始化:

```bash
cd /Users/yangyinan.8/Desktop/interview

# 1. 初始化 git

# 2. 加 .gitignore
cat > .gitignore <<'EOF'
# Python

.venv/
venv/
.env
.env.local

# Node
node_modules/
dist/
.vite/

# 编辑器
.vscode/
.idea/
.DS_Store

# 日志
*.log
logs/

# worktrees(避免误提交)
.worktrees/
EOF

# 3. 首次提交(所有现有文档)
git add background.md 产品方案.md 知识库.md .gitignore mvp/docs/
git commit -m "init: 项目文档与设计规范"

# 4. 创建两条 feature 分支(从 main 切出)
git branch feature/frontend
git branch feature/backend

# 5. 创建 worktrees 目录(不进 git)

# 6. 把两条分支检出到独立 worktree
git worktree add .worktrees/frontend feature/frontend
git worktree add .worktrees/backend  feature/backend

# 验证
git worktree list
# 应该看到:
#   /Users/.../interview                main
#   /Users/.../interview/.worktrees/frontend  feature/frontend
#   /Users/.../interview/.worktrees/backend   feature/backend
```

完成后:
- 主目录 `interview/` 留作"联调和文档中心"
- `interview/.worktrees/frontend/` 是前端 agent 的工作目录
- `interview/.worktrees/backend/` 是后端 agent 的工作目录
- 两个 worktree 是**完全独立的目录**,但共享同一个 `.git`

## 4. 联调流程(双方完成后)

### 4.1 把两条分支合到主目录

```bash
cd /Users/yangyinan.8/Desktop/interview

# 1. 切回 main 分支
git checkout main

# 2. 把前端的成果合进来
git merge feature/frontend --no-ff -m "feat: 合并前端实现"

# 3. 把后端的成果合进来
git merge feature/backend --no-ff -m "feat: 合并后端实现"

# 现在 mvp/frontend 和 mvp/backend 都在主目录里
```

### 4.2 联调启动

**开两个终端**:

终端 1(后端):
```bash
cd /Users/yangyinan.8/Desktop/interview/mvp/backend
# 激活虚拟环境、装依赖
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env  # 填 API key
uvicorn src.main:app --port 8000 --reload
```

终端 2(前端):
```bash
cd /Users/yangyinan.8/Desktop/interview/mvp/frontend
npm install
# .env.local 设置 VITE_API_BASE=http://localhost:8000 VITE_USE_MOCK=false
npm run dev
```

打开浏览器 `http://localhost:5173`,按 [接口规范.md §7](./接口规范.md) 的 Checklist 逐项验证。

### 4.3 出问题怎么办

| 现象 | 排查方向 |
|---|---|
| 前端报 CORS 错误 | 后端 CORS 配置漏了 `http://localhost:5173` |
| SSE 连不上 | 检查响应头 `Content-Type: text/event-stream`、`Cache-Control: no-cache` |
| 字段对不上(undefined / KeyError) | 两边对照 [接口规范.md §4](./接口规范.md) schema,找 diff |
| 事件名拼错 | 两边对照 [接口规范.md §3.1](./接口规范.md) 总表 |
| 打字机效果不流畅 | 后端 delta 推送间隔太短(<10ms)或太长(>200ms) |
| 满意度彩蛋不触发 | 检查 satisfaction_update 事件的 previous_value 字段是否正确 |

## 5. Worktree 操作速查

```bash
# 查看所有 worktree
git worktree list

# 新增 worktree
git worktree add .worktrees/<name> <branch>

# 移除 worktree(分支保留)
git worktree remove .worktrees/<name>

# 强制移除(有未提交修改时)
git worktree remove --force .worktrees/<name>

# 把某分支拉回主目录(合并)
git checkout main
git merge feature/<name>

# 删除已合并的分支
git branch -d feature/<name>
```

## 6. 常见疑问 FAQ

**Q1: 为什么不直接两个目录里跑,要搞 worktree?**
- worktree 共享同一个 `.git`,合并简单
- 物理隔离防止 agent 串改对方代码
- 主目录留作"中立观察区",联调时清爽

**Q2: 两个 agent 能同时跑吗?会冲突吗?**
- 能。它们各自分支独立,只要不动 `mvp/docs/`,就不会冲突
- 主目录上跑 `git status` 不会显示两个 worktree 的修改

**Q3: 如果文档需要改怎么办?**
- 由主操盘人(你)在主目录改 `mvp/docs/`
- 改完 commit 到 main
- 两个 worktree 各自 `git merge main` 拉新文档

**Q4: 我能在 worktree 里 commit 吗?**
- 必须 commit。agent 完成一块功能就 commit 一次,主操盘人才能 review

**Q5: agent 写了 bug 我想回滚怎么办?**
- 在 worktree 里 `git log` 找到上一个好 commit
- `git reset --hard <commit>` 回滚
- **小心**:这会丢弃未提交的修改

**Q6: 文档冲突了怎么办?**
- agent 不允许动 `mvp/docs/`,所以不会冲突
- 但前后端在 `接口规范.md` 上有歧义时,**必须先由主操盘人定**,然后两边各自更新代码

---

## 7. 终极原则

1. **接口规范是合同**,任何一方都不能擅自改
2. **有歧义先问,不要赌**
3. **小步 commit**,方便回滚和 review
4. **Mock 优先**,前端不依赖后端起步
5. **联调前先各自跑通**,不要把"集成才发现的问题"留到联调
